## Goal

优先把 relay 的 `web`（Svelte PWA）交互做顺畅，重点对齐 hapi 的：

1) **先规约再渲染**（reducer/reconcile，减少抖动与重复渲染）
2) **可操作卡片闭环**（permission/awaiting/tool 在卡片内完成 approve/deny/input）
3) **ready gate**（避免“看似可输入但实际未就绪”的体验）
4) **输出性能治理**（buffer 裁剪、自动滚动控制、搜索体验）

## Non-goals

- 不切换 relay 的整体传输架构到 Socket.IO / SSE（保持现有 WS fan-out + HTTP API）
- 不在 v1 引入复杂的“终端注入/语义 flush/规则引擎”（另立变更）
- 不做大范围 UI 视觉重设计（以交互/性能/结构化为主）

## Scope

范围以 `web/` 交互与渲染体验为主，但允许为满足 structured-first、事件视图 output gating、启动页 host-aware 能力与 `opencode` structured 稳定性而扩展到 `server/`、`hostd/`、`cli/` 和相关协议/文档。

### In-scope（v1）

- `web/`：
  - 从 `web/src/App.svelte` 抽离：数据规约层（blocks）、渲染层（cards）、输入与审批交互层
  - 输出视图性能：裁剪、自动滚动、搜索
  - structured-first：移动端默认“事件（卡片流）”而非“终端”

### Out-of-scope（v1）

- 与本次体验闭环无关的更大协议/架构重构；仅允许最小必要的 `include_output` / `model` / host info 元信息扩展
- 引入新的持久化模型（仍以 server `events` 回放与 web 本地状态为主）

## Plan

按“规约层 → 卡片闭环 → 性能治理 → 结构拆分”的顺序渐进式落地，优先保证行为兼容与可回滚。

- [ ] Phase 0：基线与定位（可复现路径 + 证据落盘）
- [ ] Phase 1：建立 blocks 规约层（reduce/reconcile）并以影子渲染接入
- [ ] Phase 2：可操作卡片体系（Permission/Awaiting/Tool）卡片内闭环
- [ ] Phase 3：输出视图治理（裁剪/自动滚动/搜索）
- [ ] Phase 4：拆分 `App.svelte`（panes + store）

### Phase 0：基线与定位（不改行为）

1. 记录当前 PWA 痛点与可复现路径（至少 3 条）：
   - 例：滚动抖动、卡片/弹窗切换不稳定、输入响应慢、输出过多导致卡顿
2. 在 `changes/pwa-interaction-v1/evidence/` 落盘基线/验证证据：
   - 简短屏录/截图（如可行）或文字步骤 + 浏览器 performance 简表

### Phase 1：建立“规约层”（hapi reducer/reconcile 思路，Svelte 版本）

目标：把原始事件流（messages + pending/awaiting 状态）转换成稳定的 `UiBlock[]`，并通过 key 复用避免重渲染抖动。

1. 新增目录：`web/src/lib/blocks/`
2. 定义 `UiBlock`：
   - `TextBlock`（assistant text/markdown）
   - `ToolBlock`（tool.call/tool.result，含状态与可展开详情）
   - `PermissionBlock`（run.permission_requested）
   - `AwaitingInputBlock`（run.awaiting_input）
   - `OutputTailBlock`（run.output 的 tail 摘要，仅在需要时显示）
3. 实现 `reduceToBlocks(messages, runMeta, liveAwaitingState) -> UiBlock[]`
4. 实现 `reconcileBlocks(prev, next) -> UiBlock[]`：按 `block.id` 复用对象/children（降低渲染 churn）
5. 在 `App.svelte` 中先以“影子渲染”方式接入（不改变现有交互，只替换渲染数据源），确保行为一致。

### Phase 2：可操作卡片体系（对齐 REQUIREMENTS 的 footer 交互）

1. 新增 `web/src/lib/components/cards/`：
   - `ToolCard.svelte`
   - `PermissionCard.svelte`
   - `AwaitingInputCard.svelte`
2. 卡片内闭环：
   - `PermissionCard` footer：approve/deny；支持展开查看 `op_args_summary`
   - `AwaitingInputCard` footer：输入框（Enter 发送 / Shift+Enter 换行）+ 快捷按钮（y/n/continue）
3. 保留现有弹窗作为兜底，但默认优先卡片内操作（移动端减少跳转/遮挡）。

### Phase 3：输出性能与交互治理（对齐 REQUIREMENTS）

1. 输出裁剪策略：
   - `visibleLines = floor(viewHeight / lineHeight)`
   - `bufferLines = clamp(visibleLines * 4, 200, 2000)`
2. 自动滚动控制：
   - 显式“暂停/继续”“跳到最新”按钮
   - 暂停后 10 秒无动作自动恢复
3. 输出搜索：高亮命中、上一处/下一处、命中计数
4. 事件视图默认不展示海量 `run.output`（避免淹没结构化卡片）；仅保留必要 tail。

### Phase 4：结构拆分（降低 `App.svelte` 复杂度）

目标：把“连接/数据/渲染/交互”拆开，避免单文件爆炸导致的交互不稳定。

1. 拆出模块/组件：
   - `web/src/lib/stores/session-store.ts`（runs/hosts/messages/awaiting 状态）
   - `web/src/lib/components/RunListPane.svelte`
   - `web/src/lib/components/RunDetailPane.svelte`
   - `web/src/lib/components/BlocksRenderer.svelte`
2. 保持路由与视觉结构不变（先拆逻辑再迭代 UI）。

### 手动交互验收（最小集）

1. `cd web && bun run dev` 启动 PWA
2. 登录后：runs 列表滚动流畅；选择 run 切换无明显卡顿
3. 移动端（<=640）：默认进入“事件（卡片流）”，不会自动进入“终端”
4. 待审批：PermissionCard 可在卡片内 approve/deny，且回放一致
5. 待输入：AwaitingInputCard 可在卡片内输入并发送（Enter/Shift+Enter 正常）
6. 输出：暂停/继续/跳到最新可用；长输出不持续增长到卡死

### 回归（不影响后端）

```bash
bash scripts/e2e.sh
```

## Risks & Rollback

- 风险：`App.svelte` 拆分与数据规约引入回归（交互断裂、卡片状态错乱）。
- 缓解：
  - 分阶段落地：Phase 1 影子接入 + 行为不变；通过后再替换渲染
  - 关键行为（approve/deny/input）保留旧路径作为兜底
- 回滚：
  - 只回滚 `web/` 相关提交；或临时切回旧渲染路径（保留 feature flag/开关）
